## Cabling

- **Action**: Inspect all cabling.
- **Spec**: No visible damage present.
- **Result**:
- **Status**:
- **Comment**:

### Procedure Details (Verbatim from Official Checklist)

Check and correct as necessary:  
- All **system cables, power and PE cables, and hoses** for dirt and damage.  
- Inspect **for wear or loose connections.**  
- Ensure that **cables are secured safely.**  


### Procedure Details Revision

Inspect all external and internal cabling for physical integrity and security.  
- Look for any **bent, pinched, or overly stretched cables** that may cause signal degradation or power issues.  
- Check that all **connectors are securely seated** and show no signs of corrosion or damage.  
- Verify that **shielding and insulation are intact**, especially on high-frequency or power cables.  
- Ensure all cables are properly routed, avoiding stress points and excessive bending.  
- Confirm that grounding and power connections comply with safety and **EMI shielding standards**.  

If **no visible damage is found**, and all cables appear secure, mark **Pass**.  
If **minor wear is detected but does not affect function**, note it under **Other** and specify details in the **Comment** field.  
If **cabling failure directly affects system performance**, mark **Fail** and recommend immediate action.  

(Note: Result, Status, and Comment fields are blank by default until filled in by automation or the engineer.)

## System

- **Action**: Verify system functionality.
- **Spec**: No errors on startup.
- **Result**:
- **Status**:
- **Comment**:

### Procedure Details (Verbatim from Official Checklist)

Check and correct as necessary:  
- **PC (operating system) and application software start without errors** (check error log).  
- **System corresponds to the installation requirements**.  

### Procedure Details Revision

Perform a system-wide check to ensure stable operation and compliance with installation requirements.  
- Verify that the **PC and operating system boot successfully** without unexpected errors.  
- Check that **all required services and background processes** start correctly.  
- Open the **application software** and confirm it runs without crashes or missing dependencies.  
- **Review the system error log** for any warnings or failures that might indicate potential issues.  
- Ensure that the **system configuration matches the installation requirements**, including network settings, hardware connections, and software versions.  

If **no errors are detected and the system meets installation requirements**, mark **Pass**.  
If **minor errors are logged but do not impact functionality**, mark **Other** and specify details in the **Comment** field.  
If **critical errors prevent the system from operating correctly**, mark **Fail** and escalate for troubleshooting.

# Platform Governance Checks

## Credential Safety

- **Action**: Validate credential management
- **Spec**: Complies with the Cvmewt Secure Access Standard
- **Result**:
- **Status**:
- **Comment**:

### Procedure Details (Reference Implementation)

Review credential rotation logs and confirm that service principals, API tokens, and local `.env` files match the retention policy defined in `docs/design/runtime_consolidation_plan.md`.  
- Ensure **no shared secrets** are checked into `docs/checklists/` or any asset-pack bundle.  
- Verify that **automation accounts** use scoped tokens and expire within the allowed window.  
- Confirm that **break-glass credentials** are sealed in the vault and referenced only by incident runbooks.  
- Capture the **Secret Rotation ID** and timestamp if a rotation was triggered during the visit.

### Procedure Details Revision

If every checklist workspace passes the credential audit, mark **Pass**.  
If a minor issue exists (e.g., a token scheduled for rotation later the same day), use **Other** and describe the mitigation.  
If a credential exposure is detected, mark **Fail**, revoke the token immediately, and escalate to `security@cvmewt.com`.

---

## Data Residency & Telemetry Privacy

- **Action**: Inspect data residency posture
- **Spec**: Meets tenant residency declaration and APIM telemetry policy
- **Result**:
- **Status**:
- **Comment**:

### Procedure Details (Reference Implementation)

Check that **workspace exports, checklists, and TEX reports** are stored under the tenant-owned path (S3 bucket, Azure container, etc.) defined in the deployment manifest.  
- Ensure **no generated TEX/PDF artifacts** remain inside the repository or shared build cache.  
- Validate that **telemetry forwarding** (Application Insights, CloudWatch, etc.) is limited to aggregated metrics that cannot leak proprietary checklist data.  
- Document the **region + storage identifier** used for the export so compliance teams can reconcile audits.

### Procedure Details Revision

Mark **Pass** when storage locations align with the residency declaration and telemetry only contains aggregated health metrics.  
Use **Other** when alignment is pending (for example, exports temporarily stored in a staging bucket awaiting transfer).  
Mark **Fail** if proprietary checklist data is found outside the tenant-approved boundary and initiate the incident response procedure in `docs/design/runtime_consolidation_plan.md`.
